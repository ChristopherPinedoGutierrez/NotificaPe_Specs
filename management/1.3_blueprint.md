# BLUEPRINT DE SOLUCIÓN TÉCNICA

## ARQUITECTURA DE DATOS Y ENTIDADES

Se utilizará Supabase (PostgreSQL) como infraestructura principal (BaaS). El modelo de datos ha sido estructurado respetando la convención de nomenclatura `EntidadXEntidad` y la separación de catálogos requerida por el equipo de desarrollo, pero garantizando la resiliencia offline mediante el uso de UUIDs transaccionales.

**Nota sobre Claves Primarias (UUID vs BIGINT):** Para prevenir colisiones (Foreign Key constraint fails) durante la sincronización de bases de datos locales (SQLite en modo offline) hacia la nube, todas las entidades transaccionales y de identidad utilizan UUID. Para cumplir con los requisitos de paginación por cursores del equipo (Keyset Pagination), se han agregado columnas secundarias "SecuenciaLocal" de tipo BIGINT autoincremental de solo lectura.

### Propósito de Cambios Críticos y Alineación

- **Extracción Numérica y Privacidad:** La tabla `NotificacionesXDispositivo` incorpora MontoCentimos, Moneda y Privada, protegiendo el rendimiento de la BD frente a extracciones Regex en tiempo de consulta y resguardando las finanzas personales del Contratante desde el origen.
- **Modelo Prepago e Idempotencia:** Se abandona SaaS clásico por un Inventario Prepago ligado a `OrdenesCompra`, usando el número de orden externo como Primary Key para impedir matemáticamente el fraude de doble canje.
- **Estandarización Internacional:** Eliminación de ENUMs restrictivos en favor del formato ISO 4217 (Monedas) y E.164 (Teléfonos) para expansión sin migraciones de base de datos.
- **Flexibilidad de Cobro:** Separación de la imagen del QR (`BilleterasXDispositivo`) del número de teléfono principal (`DispositivosXContratante`) para soportar múltiples billeteras con distintas imágenes sobre un mismo terminal físico.

## FLUJO LÓGICO Y ENDPOINTS DE API

### Procesos Internos y Conexiones

- **Validación Prepago:** Web Panel envía Orden -> Edge Function de Supabase hace GET a API externa (WooCommerce/Shopify) -> Si es válido y el email coincide, hace INSERT seguro.
- **Captura y Extracción (App Admin):** El NotificationListenerService captura la alerta. Prueba en bucle las reglas Regex (FiltrosXBilletera) para extraer el Monto localmente antes de insertar en SQLite/Supabase.
- **Sincronización y Recuperación:** La App Viewer solicita el historial de la caja basada en el límite de paginación (SecuenciaLocal), estableciendo luego conexión Supabase Realtime (WebSockets) para el vivo.
- **Gestión de Disputas (Epic 5):** La inserción en `NotificacionesAUsuarios` actúa como reclamo de la transacción. Un trigger en la base de datos gestiona la competencia: si hay un solo reclamante, gana (`APROBADO`); si hay más de uno, el estado de la notificación maestra pasa a `REVISION` para mediación del administrador.

## STACK TECNOLÓGICO Y JUSTIFICACIÓN

- **Kotlin (Android Nativo):** Obligatorio para NotificationListenerService.
- **Next.js / TypeScript:** Framework para el Web Panel.
- **Supabase (PostgreSQL + Auth + Realtime + Storage):** Acelera el desarrollo del MVP, aloja las imágenes QR, evita desarrollar un backend robusto de cero y usa `Functions` nativas de PostgreSQL para cálculos matemáticos precisos (Ceil Rounding en upgrades de licencias).

## MANEJO DE CASOS CRÍTICOS Y SEGURIDAD

- **Interrupciones de Red:** Las UUIDs aseguran que los datos guardados offline (Notificaciones y Asignaciones) no colisionen al subir en bloque al recuperar la conexión.
- **Actualización Dinámica de Billeteras:** Las reglas de lectura de SMS son acumulativas en BD. Android itera sobre ellas, previniendo fallos masivos si un banco actualiza el texto de su aplicación.
- **Fallos de Hardware (Cámaras):** Integración de `CodigoAcceso` manual como plan de respaldo ante fallos en la lectura de códigos QR.

## ARQUITECTURA CLIENTE: EMISOR (APP ADMIN)

La aplicación emisora basa su resiliencia en tres pilares arquitectónicos exclusivos (Auditoría App Admin):
1. **Captura Offline-First (Room):** El `NotificationReceiverService` (Foreground) captura los pagos y los persiste localmente en SQLite/Room con estado `PENDING` antes de intentar subirlos a Supabase. Un `SyncWorker` gestiona reintentos en background.
2. **Ciclo de Vida Persistente (Centinela):** El servicio sobrevive al "Swipe to close" (onTaskRemoved), se inicia tempranamente al elegir caja, y se destruye solo si la sesión se invalida en la Zona Neutral (Mis Vinculaciones).
3. **Bloqueos de UI Estrictos (Overlays):** Se restringe la UI mediante capas elevadas si la conexión se pierde (`FullSyncingOverlay`), faltan permisos, o si el PDV se desactiva.
4. **Separación de Identidad y Sesión:** Se eliminan dependencias globales para transacciones; la identidad se inyecta por argumentos de navegación para prevenir carreras de datos en redes lentas.

### [CR-002] Autoarranque Resiliente ante Reinicios (Boot Receiver) en App Admin
- **Fecha:** 2026-07-13
- **Justificación:** Garantizar alta disponibilidad 24/7 en la terminal emisora de cobros. Actualmente, ante reinicios accidentales o actualizaciones del sistema, el monitoreo no inicia por completo (permaneciendo inactivo) hasta que el usuario abre manualmente la aplicación. Se requiere levantar automáticamente el Foreground Service del Listener tras el encendido/desbloqueo del teléfono.
- **Impacto Técnico:**
  - **Permisos:** Declarar `android.permission.RECEIVE_BOOT_COMPLETED` en el archivo `AndroidManifest.xml` de la App Admin.
  - **Componente Nuevo:** Crear `com.notificape.admin.service.BootReceiver` extendiendo de `BroadcastReceiver` para capturar las intenciones `BOOT_COMPLETED` y `QUICKBOOT_POWERON`.
  - **Lanzamiento de Servicio:** En el receptor, si el intento coincide con las acciones del boot, iniciar `NotificationReceiverService` usando `context.startForegroundService(intent)`.
  - **Resiliencia de Red:** Mantener la lógica de inyección de Hilt y corrutinas de resiliencia activas en el servicio, las cuales arrancan autónomamente al procesar el `onStartCommand` del servicio sin requerir el hilo visual de la UI.
  - **Cero Afectación:** Cero impacto en base de datos PostgreSQL, esquemas relacionales o endpoints del Web Panel.

### [CR-003] Mapeo Detallado de Excepciones de Credential Manager en App Viewer
- **Fecha:** 2026-07-14
- **Justificación:** Necesidad de diagnosticar remotamente fallos en el inicio de sesión con Google reportados por testers en dispositivos de terceros (como Honor/Magic OS o Android 14/16). Al no tener acceso directo al hardware del usuario, un mensaje genérico ("Autenticación cancelada") enmascara fallos de configuración de autocompletar o GMS, por lo que se requiere detallar el tipo de excepción arrojado por el SDK.
- **Impacto Técnico:**
  - **Excepciones Segmentadas:** Modificación del catch de `GetCredentialException` en `LoginScreen.kt` para interceptar `GetCredentialCancellationException`, `NoCredentialException`, `GetCredentialInterruptedException` y `GetCredentialProviderConfigurationException`.
  - **Mensajes UI:** Mostrar en la pantalla de error información técnica útil para el usuario final o tester sobre cómo solventar el fallo del sistema.
  - **Cero Afectación:** No afecta la base de datos relacional de Supabase ni la funcionalidad de otras aplicaciones del ecosistema.

### [CR-004] Resiliencia de Conexión Realtime y Delta Sync en Viewer
- **Fecha:** 2026-07-14
- **Justificación:** Corregir la debilidad de la reconexión asíncrona de Realtime al volver de segundo plano o tras transiciones de red física (Modo Avión). El socket de Supabase suele desconectarse y dejar canales en estado "zombi" o inútiles en memoria. Se requiere procesar transiciones de forma secuencial mediante un Mutex y gatillar la re-suscripción activa de canales tras la reconexión exitosa del socket.
- **Impacto Técnico:**
  - **Clonación de NetworkMonitor:** Introducir `NetworkMonitor` en Viewer para detectar en caliente el restablecimiento de red del dispositivo.
  - **Desconexión Preventiva:** Desconectar preventivamente el socket de Supabase al detectar `isOnline == false` para evitar discrepancias de estados en la librería interna.
  - **Reset Atómico (Mutex):** Introducir `profileMutex` en `RealtimeCoordinator` y refactorizar `setProfile()` y `hardReset()` para ejecutarse de forma secuencial y bloqueante, previniendo condiciones de carrera concurrentes.
  - **Re-suscripción Activa:** Observar la transición a `CONNECTED` del socket para forzar automáticamente un `hardReset()` y reconstruir los canales activos, evitando que se queden en `none`.
  - **Acoplamiento de Delta Sync:** Gatillar `syncScavenger.performDeltaSync` únicamente tras el estado `SUBSCRIBED` del canal de notificaciones, usando un delta de tiempo calculado dinámicamente según la inactividad en background.
  - **Cero Afectación:** No afecta la base de datos relacional de Supabase ni interfiere con las aplicaciones Admin o Web.

### [CR-005] Flujo de Registro e Ingreso Manual Alternativo en App Viewer
- **Fecha:** 2026-07-16
- **Justificación:** Prevenir el bloqueo de testers/usuarios que utilicen dispositivos móviles sin GMS (Google Mobile Services) o con firmas SHA-1 no registradas (como compilaciones de depuración locales). Se propone diseñar una vía alternativa de autenticación basada en correo/contraseña convencional, con su respectiva UI de registro, vinculación manual a cajas y gestión en Supabase.
- **Impacto Técnico:**
  - **Autenticación Clásica:** Configurar el flujo de correo/contraseña (Email Auth) en Supabase.
  - **Vistas en Viewer:** Diseñar pantallas de Login manual y Registro de cuenta en la aplicación.
  - **Soporte de Billeteras:** Implementar interfaz para asignación manual de billeteras y alias al no contar con el auto-mapping del perfil de Google.
  - **Cero Afectación:** No altera las funcionalidades existentes basadas en Credential Manager de Google ID, operando como una opción secundaria e independiente.
